SP107 — mono samples play centered, not left-channel-only diagnosis → fix → verification · PR #415

A real-example parity investigation. monday_blues read with a large spectral diff (L2 mel-dB 31.2, MFCC 260) on kick-dominated material. The pre-written hypothesis was a "sub-bass-deficient kick" — a gain-vs-rendering fork. Both candidates were refuted. The real cause was a one-axis-missing player-selection bug: mono samples rendered to the left channel only, the right channel silent. Read top-to-bottom for the chronology.
Question — why does kick-dominated material (:drum_heavy_kick) read a large spectral diff vs desktop, with no wrong notes?
Resolved Because selectSamplePlayer chose the sample player by opt-complexity alone and never by the sample's channel count. A 1-channel (mono) buffer routed through basic_stereo_player (which expects a 2-channel buffer) left the right output channel silent. When the comparator mono-averages (L+R)/2, a left-only kick is halved relative to centered synth layers — collapsing the sub-band share and inflating MFCC. Fixed by adding the channel-count axis, mirroring Desktop SP's resolve_specific_sampler.
TL;DR
  1. The bug: mono samples were left-channel-only on web. Kick A/B: web L=0.109 / R=0.000 vs desktop centered L=R=0.2265.
  2. The measurement trap: the prior session's "0.24× attenuated kick" was an artifact of mono-averaging a left-only signal. Reading L and R separately revealed R=0 — a channel bug masquerading as a gain bug.
  3. Root cause: selectSamplePlayer(opts) branched only on opt-complexity; it never read the sample's channel count. Desktop's resolve_specific_sampler(num_chans, args_h) (sound.rb:3470-3478) picks basic_mono_player for 1-channel samples (which Pan2-centers).
  4. The fix (PR #415): add the channel-count axis — cache audioBuffer.numberOfChannels from the existing decode, resolve it before player selection, use mono players for mono samples. Pure synthdef-name swap; zero param-translation change; defaults to stereo (no regression).
  5. Verified Level-3: web kick R-channel 0.000 → 0.1090 (centered, deterministic); full monday_blues L2 31.2→14.0, MFCC 260→119; launch gate stays 5/7 = 71.4% PASS.

The measurement trap — why the first read was wrong

The prior session isolated the kick and reported it "0.24× attenuated / sub-bass-deficient," pointing at gain or sample-player filtering. That number was an artifact of the analysis, not the engine:
  • Symptom mismatch: the comparison tool reported web kick peak 0.109, but a per-hit script (which mono-averaged (L+R)/2) reported 0.0545 — a clean 2× disagreement on the same file.
  • The witness check: two measurements of the same WAV cannot disagree by 2× unless one is wrong. Reading L and R separately showed L=0.109, R=0.0000 — the mono-average had halved a single-populated channel.
  • Control 1: the mod_saw synth alone is L=R=0.1514 (centered) → the defect is sample-specific, not global.
  • Control 2: the kick at rate: 1.0 is also left-only → not a rate: artifact.
  • Per-channel truth: the kick's per-channel gain (0.48×) and spectral shape (99% sub <80 Hz) match desktop exactly. The only defect is the missing right channel.
The lesson (now vyapti SV54): on stereo WAVs, read L and R separately before any mono-collapse — a channel-routing bug otherwise hides as a "gain" bug.

Root cause — one axis missing vs the reference

Desktop Sonic Pi selects the sample player on two independent axes; our engine had only one.

Desktop SP — sound.rb:3470-3478

def resolve_specific_sampler(num_chans, args_h)
  if complex_sampler_args?(args_h)
    (num_chans == 1) ? :mono_player : :stereo_player
  else
    (num_chans == 1) ? :basic_mono_player : :basic_stereo_player
  end
end
# num_chans read from the loaded buffer (buf_info.num_chans, :3498)
# BEFORE the player is resolved → mono samples get a mono player

Ours — before the fix (SoundLayer.ts)

function selectSamplePlayer(opts) {
  // opt-complexity axis ONLY — channel count never read
  if (anyComplexOpt(opts)) return '…stereo_player'
  return '…basic_stereo_player'  // ← mono buffer → R silent
}
A mono buffer played through basic_stereo_player (which reads a 2-channel buffer) writes audio to the left bus channel and leaves the right empty. The mono players (basic_mono_player/mono_player) Pan2-center a 1-channel PlayBuf to both channels. The basic mono and basic stereo players share identical arg signatures (synthinfo.rb:5013, BasicStereoPlayer < BasicMonoPlayer) — so the fix is a pure synthdef-name swap.

The fix — add the channel-count axis (PR #415)

  • Selection: selectSamplePlayer(opts, numChans = 2) → basic_mono_player/mono_player when numChans === 1; stereo variants otherwise. numChans defaults to 2, so callers that omit it keep today's behavior (no regression).
  • Ordering: SuperSonicBridge caches audioBuffer.numberOfChannels from the existing decodeAudioData (new sampleChannels map) and resolves it before player selection — matching desktop's buf_info.num_chans-known-before-resolve ordering (sound.rb:3496-3498).
  • Synthdefs: mono players lazy-load via the existing ensureSynthDefLoaded (both confirmed in CDN @0.57.0).
  • Failure mode: on decode failure, falls back to stereo — degrades to pre-fix behavior, never to no-audio.
  • Self-review: a follow-up commit clears sampleChannels in all three sample-teardown sites (SV38 hygiene), matching the sibling cache maps.

Full code, end to end — how the engine runs this composition

The fix is one line of selection logic, but it sits inside a full pipeline. Here is the exact monday_blues source and how every construct in it flows from Ruby text to scsynth audio — each claim below is read straight from the web engine's event log for this run (Level 2 inference), with the audio confirmed at Level 3 above. 
use_debug false

live_loop :drums do
  puts "slow drums"
  6.times do
    sample :drum_heavy_kick, rate: 0.8
    sleep 0.5
  end
  puts "fast drums"
  8.times do
    sample :drum_heavy_kick, rate: 0.8
    sleep 0.125
  end
end

live_loop :synths do
  use_synth :mod_saw
  use_synth_defaults amp: 0.5, attack: 0, sustain: 1, release: 0.25, mod_range: 12, mod_phase: 0.5, mod_invert_wave: 1
  notes = (ring :F, :C, :D, :D, :G, :C, :D, :D)
  notes.each do |n|
    tick
    play note(n, octave: 1), cutoff: (line 90, 130, steps: 16).look
    play note(n, octave: 2), cutoff: (line 90, 130, steps: 32).look
    sleep 1
  end
end

live_loop :snare do
  sample :drum_snare_soft
  sleep 1
end

The pipeline — Ruby text → audio

  • 1 · Transpile Transpiler.ts — tree-sitter partial fold over the Sonic-Pi Ruby subset turns the source into JS. 6.times do…end, notes.each do |n|…end, blocks, and the (ring …) / (line …) paren-call forms are all recognised; bare notes = … is emitted as a bare assignment so the sandbox Proxy can capture it. Falls back to the regex transpiler if tree-sitter fails.
  • 2 · Sandbox createIsolatedExecutor() — user JS runs inside with(__scope__); the Proxy's has() returns true for everything so notes = … writes into scope-isolated storage. use_synth / use_synth_defaults set thread-local state read by later play calls.
  • 3 · ProgramBuilder builds Step[] (the free monad) — one live_loop registration per loop; inside, each sample / play / sleep is a Step. tick/look advance and read a per-loop counter; (line 90,130,steps:N).look resolves to a concrete cutoff at build time per iteration.
  • 4 · VirtualTimeScheduler — the core innovation: each sleep returns a Promise that only the scheduler resolves. The three live_loops run as cooperative coroutines in virtual time — drums advancing 0.5/0.125 beats, synths 1 beat, snare 1 beat — interleaved deterministically, no thread blocking.
  • 5 · AudioInterpreter walks the Step[] and dispatches each play/sample to SuperSonicBridge at its virtual-time-mapped audio time.
  • 6 · SoundLayer normalises params: note(:F, octave:1)→MIDI 29 (the #409 fix), symbol resolution, cutoff kept for synths / aliased to lpf for samples, BPM time-scaling. It also decides nothing about channels — that is the player layer (the SP107 fix).
  • 7 · SuperSonicBridge selectSamplePlayer(opts, numChans) — reads the cached channel count and routes each sample to its player. Both :drum_heavy_kick and :drum_snare_soft are mono → basic_mono_player (post-fix, centered). Emits /s_new to the per-loop track bus.
  • 8 · scsynth (WASM) renders each node; per-loop track buses (16/18/20) mix to the master and out to Web Audio.
