:lpf — desktop scsynth vs web WASM scsynth raw-OSC + DSL + FX sweep + e2e

This page started as a raw-OSC parity test but has grown to cover the full investigation — raw-OSC (passes 1–3 + sub-tests), DSL path with engine fixes (third pass), the WASM gain-deficit characterization (fourth pass), the post-fix FX-sweep + e2e regression (fifth pass), and the rhythm-parity confirmation on the lpf inspector view (sixth pass). Read top-to-bottom for the chronology, or jump to a section via the TL;DR below.
Question — does WASM scsynth produce the same audio as native scsynth for identical OSC?
TL;DR (post-2026-05-09 evening):

  1. The WASM scsynth has a uniform ~0.6× output deficit vs native for the same byte-identical synthdef, applied to every signal regardless of FX or cutoff. Verified across :level passthrough, :lpf MIDI 30–127, :hpf MIDI 30–127, :bpf. Synthdef SHA-256 byte-match. Filable upstream at samaaron/supersonic.
  2. SP72 (use_bpm leak in nested live_loop) is fixed. Web wallclock cadence is now exact 4 Hz on the lpf snippet (use_bpm 120 + sleep 0.5). Confirmed by no-FX A/B in Sixth pass.
  3. MIXER.AMP raised 1.2 → 3 as empirical compensation for the WASM deficit. Effective gain ~0.9× of native, RMS parity within ~7%. Tau-aligned 1.2 was the old setting; AMP=3 is today's.
  4. Full FX sweep + e2e regression with both fixes (Fifth pass): mean FX-sweep score 37.9 → 60.2; 27 of 40 FX upgraded tier; 8 of 10 e2e fixtures within 0.62–0.88× RMS of desktop. Two outliers (02_fx_chain, 07_ambient) belong to a separate non-constant-gain class — feedback-loop FX (reverb/echo/chorus) in WASM render at different gain than filter UGens.
  5. The "rhythm looks/sounds broken on lpf" appearance is real and audible. Wallclock scheduling IS identical (4 Hz both — confirmed via web engine OSC trace at 250 ms intervals, no-FX parity, and Sam-chain autocorr). But native LPF's louder ringing tails create audible secondary peaks between scheduled hits, so listeners hear desktop with :lpf as faster/denser (~61 hits/sec) than web (~42 hits/sec) — same scheduled tempo, different perceived density. Same upstream root cause (WASM-vs-native UGen rendering) as the gain deficit, expressed in a third observable axis. See Sixth pass correction box.
  6. What the FX layer adds beyond no-FX base separates into 4 structural classes (Seventh pass): linear filter (well-behaved, AMP=3 fixes them), non-linear (input-sensitive, AMP=3 partially helps), modulation (LFO-precision divergence), feedback delay (exponentially compounding, source of the 07_ambient 2.96× and 02_fx_chain 0.11× e2e outliers).

What this tells us, directly observed:
Two configurations on web — fundamentally different routing topologies (target=fx-group vs root, in_bus=16 vs 64, out_bus=0 vs 14) — produce audio that matches at 4 decimal places (0.0741 / 0.0145). Addressing fields are routing labels. They tell scsynth WHERE to write, not WHAT to compute. The DSP is determined by the synthdef + buffer + params, all held constant across configurations.

And neither web configuration reaches desktop's 0.6244 peak. The 0.119× gap is in the WASM ugens themselves — specifically, exp-010's no-FX raw test (no fx_lpf, just BufRd + Out) showed the same 0.23× peak ratio, localizing the divergence to basic_stereo_player's playback ugens, which are upstream of any FX wrapping. Confirmed by Lokayata, not inferred.
Update 2026-05-09 (refines the conclusion above): the deficit is NOT specific to basic_stereo_player. The Fourth pass ran the same MIDI-cutoff sweep on :lpf, :hpf, :bpf, and a no-op :level passthrough — the ratio is flat at 0.55–0.66× across every WASM signal path. So the locus is the WASM scsynth render layer itself (probably Out / AudioWorklet stage), not the playback ugens. The "exp-010 → basic_stereo_player" framing was correct evidence but a too-narrow interpretation.

Fourth pass (2026-05-09) — characterizing the WASM gain deficit

After the AMP=3 result, the question shifted from "is there a residual?" to "what shape is the residual?" The hypothesis was that the WASM basic_stereo_player's envelope generator has a different attack-time minimum than native, producing peakier transients. The data instead points to a uniform output gain deficit in the WASM scsynth itself, applied to every signal regardless of FX.

Step 1 — synthdef binary check

Byte-identical. Both engines load the exact same compiled scsyndef. So the divergence must come from the WASM rendering of identical UGens, not from a different synth graph or different parameters baked in.

Step 2 — decompile the synthdef and read the UGen graph

Tool: tools/_probe-decompile-scsyndef.py (parses SCgf v1 binary). Key UGens in sonic-pi-fx_lpf (70 ugens total, 22 params): 
# Cutoff path: param:cutoff (default 100 MIDI) → EnvGen (slide) → MIDICPS (op17) → Hz
u37 EnvGen   rate=control (param:cutoff …)
u38 UnaryOp  rate=control op=17 (MIDICPS)
# Filter (one per stereo channel) — Butterworth 2-pole, NO resonance
u39 LPF      rate=audio   (u30.input_left, u38.cutoff_hz)
u58 LPF      rate=audio   (u57.input_right, u38.cutoff_hz)
# Dry/wet crossfade per channel
u66 XFade2   rate=audio   (dry_left, wet_left, mix*2-1, amp)
u68 XFade2   rate=audio   (dry_right, wet_right, mix*2-1, amp)
u69 Out      rate=audio   (param:out_bus, u66, u68)
Two important things from the graph: (a) The filter is LPF — a 2-pole Butterworth with no resonance / Q parameter, so the divergence cannot be Q. (b) The cutoff parameter is in MIDI notes, not Hz: UnaryOpUGen op=17 (MIDICPS) converts before reaching the LPF freq input.

Step 5 — Tau context, and what AMP=3 was actually doing

Sonic Tau (a separate browser-WASM Sonic Pi project) settled on PRE_AMP=0.2 / AMP=6 (effective 1.2) for its mixer. That value was empirically calibrated to avoid clipping the WASM-quiet path on Tau's content. We adopted Tau's config as our own MIXER default because Tau had already done the calibration legwork — but Tau didn't characterize why the WASM path is quiet, just compensated for it.

Today's data fills in that missing piece: the WASM scsynth uniformly outputs ~0.6× of native, baseline, before any FX or mixer involvement. Tau's mixer setting of 1.2 was matching this by coincidence — they were compensating for the WASM deficit at the mixer layer, intuitively, without knowing the underlying constant.

Our AMP=3 setting (this session) bumps the mixer 2.5× higher than Tau-aligned 1.2. Effective gain becomes 2.5 × 0.6 = 1.5× pre-limiter. The Limiter.ar(0.99) clamps the peakiest transients, leaving the per-beat windows at ~1.07× RMS / ~1.14× peak vs desktop — the residual we measured in the third-pass section.

Implication. The 1.14× per-beat peak ratio after AMP=3 is the sum of two things: (1) the WASM scsynth's constant 0.6× output deficit (corrected by AMP=3 to ~1.5×), and (2) the mixer Limiter.ar(0.99) clamping native's peaks slightly more than web's (because web's pre-limiter peaks are right at the ceiling, native's are slightly below). It is not a per-hit envelope-shape divergence in basic_stereo_player — that hypothesis was wrong.

Step 6 — content-specific reversal (older finding, now contextualized)

Memory note feedback_wasm_gain_staging.md (2026-05-04, multi-fixture sweep) found that across 10 e2e compositions, web/desktop RMS ratios spanned 0.30× to 4.20× — three fixtures had web LOUDER. Today's flat-0.6× class signal is on raw-OSC paths through filter UGens. The 4.20× outlier (07_ambient, heavy reverb + prophet + echo) implicates time-domain feedback-loop FX (reverb, echo, chorus) as a separate class with non-constant gain behavior. Those weren't measured today.

Sixth pass (2026-05-09 evening) — "rhythm still looks broken on lpf?" — measurement artifact, not bug

Looking at /?fx=lpf in the FX inspector, the desktop and web spectrograms visually suggest different rhythms — desktop's hits look denser, beefier, and the recording-window asymmetry makes it ambiguous which side is at the right tempo. This is a measurement artifact from the same WASM-vs-native LPF gain delta we characterized above — not a tempo bug.

    Reading. Desktop fires with a ~1 s recording-start latency, so beats 0–2 land in pre-audio silence. Beats 3–7 carry the actual hits (kicks at full ~0.62 peak, snares at ~0.62 peak with lpf). Web has no equivalent latency; all 8 beats carry audio, but at ~0.12× the peak amplitude (this number is from the raw-OSC pre-mixer path; with AMP=3 in the engine path the ratio comes back to ~0.85× — see Third pass). That uniform ratio matches the raw-OSC :level, :bpf, no-FX measurements from yesterday's exp-010 and today's Fourth pass sweep: the WASM scsynth output is uniformly ~0.6× of native for byte-identical /s_new + same buffers, regardless of FX or cutoff. Below the bridge. Not our chain.

    Confound disclosure (probes). The precondition flags fire because the captured WAVs are not signal-shape comparable: desktop has the recording-start latency offset (envelope_lag ≈ 2 s), and the per-beat slicer at t = 0 lands desktop beats 0–2 in silence. Both effects are unrelated to the WASM ugen question — they're artifacts of how each side's recording bracket opens. The right way to read this page is per-hit (beats 3–7 desktop ↔ all-beats web with a time-shift), not aggregate RMS.
